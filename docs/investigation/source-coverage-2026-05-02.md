# Intelligence Brief source-coverage audit

**Date:** 2026-05-02
**Branch:** docs/source-coverage (audit findings only — no code changes)
**Trigger:** Customer asking "what sources is this brief pulling from?" and the
implicit "what's missing that a $50/mo intelligence brief should cover?" This is
the input-side parallel to the content-quality audit in `docs/brief-content-audit.md`
(PR #40).

**Note on empirical evidence:** the live source-counts in §2 require
staging-DB access; this codespace does not have credentials. SQL queries that
would ground the per-source weekly volume are listed in §6 for the operator to
run. Health-check results in §3 were generated by hitting each upstream URL
directly from the codespace on 2026-05-02 and are reproducible.

---

## 1. Executive summary

SecureLogic AI ingests from **26 distinct source slugs** spanning 23 unique
upstream URLs across two pipelines (engine direct adapters + worker RSS
collectors). On a clean health check today, **18 sources return healthy
HTTP 200**, **7 are broken** (all 7 in the worker RSS path: 6× 404 from URLs
that have moved or been retired, 1× 403 from a bot-blocked publisher), and
**1 (`regulatory_sec_8k`) works only with a proper `User-Agent` containing a
contact email** as SEC EDGAR requires — the production code uses the default
`rss-parser` UA and is likely silently 403'd in prod.

The biggest **content-side gap** is per-vendor PSIRT advisories (Microsoft
MSRC, Cisco PSIRT, Fortinet, Palo Alto, Ivanti, Citrix, etc.) — currently zero
PSIRT coverage. The most valuable adjacent sources to add, in order of
value-to-effort: **GitHub Security Advisories (GHSA)**, **Microsoft MSRC**,
**FBI/CISA joint advisories (StopRansomware feed)**. Each is one PR, none
require commercial licensing, all three are content the customer's CISO peers
get from elsewhere today.

The **operational gap** that should ship before any new source: the 7 broken
RSS URLs are 27% of the slug inventory and have been silently failing
(`fail-open` semantics — see worker `securityNewsFeed.ts:62`) for an unknown
duration. Nothing alerts on a feed disappearing. Add a one-time URL fix PR
plus a synthetic monitor before adding new sources.

## 4. Gap analysis — what a $50/mo brief should cover that we don't

Source categories a CISO buying an executive intelligence brief would expect
to see represented. "Covered" means at least one healthy source slug producing
a non-trivial weekly volume. "Partially covered" means content exists in
adjacent feeds but isn't structured as the named source. "Not covered" means
zero ingestion today.

### A. Vulnerability databases — **covered**

- ✅ `cisa_kev` — actively-exploited CVEs with federal due dates
- ✅ `nvd` — authoritative CVSS, full CVE corpus
- Effort to add another: N/A (the two market-standard sources are in)

### B. Federal/government advisories — **partially covered**

- ✅ `cisa_alerts` — CISA Cybersecurity Advisories (advisories, ICS bulletins)
- ✅ `regulatory_cisa` — duplicate of cisa_alerts via worker (slug fragmentation, see `docs/dedup-audit.md` Issue B)
- ⚠️ `regulatory_sec_8k` — broken UA, currently no signal
- ❌ FBI / IC3 (Internet Crime Complaint Center alerts)
- ❌ NSA Cybersecurity Advisories
- ❌ CISA "StopRansomware" joint advisories (separate feed from Cybersecurity Advisories — these are FBI+CISA joint, named-actor focused)
- ❌ EU CSIRTs (CERT-EU advisories, member-state CERTs)
- ❌ UK NCSC advisories
- **Effort:** ~1 PR per source (RSS or simple JSON). NSA, NCSC, FBI/IC3 are all RSS or Atom.

### C. Vendor PSIRT feeds — **NOT covered**

The single biggest substantive gap. A brief that talks about "what should you patch this week" without ingesting from the vendors who publish patches is missing the most operational input.

- ❌ Microsoft MSRC (Patch Tuesday + zero-days; ATOM feed at `api.msrc.microsoft.com/cvrf/v3.0/updates`)
- ❌ Cisco PSIRT (RSS + JSON API)
- ❌ Fortinet PSIRT (FortiGuard advisories)
- ❌ Palo Alto Networks Security Advisories
- ❌ Ivanti Security Advisories
- ❌ Citrix Security Bulletins
- ❌ VMware Security Advisories (now Broadcom)
- ❌ Adobe Security Bulletins
- ❌ Oracle Critical Patch Updates (quarterly)
- ❌ Red Hat Security Advisories (RHSA)
- ❌ SUSE / Ubuntu / Debian security trackers
- **Effort:** ~1 PR per vendor for RSS-based feeds, ~2-3 PRs for CSAF/JSON-API based. Most vendors offer RSS.
- **Note:** the dedup audit (`docs/dedup-audit.md` §3) explicitly anticipated PSIRT feeds and confirmed they're safe to add — they get their own source slugs and CVEs already discriminate them in the dedup hash.

### D. Industry / sector CERTs — **NOT covered**

- ❌ ICS-CERT (industrial control systems — separate from CISA Cybersecurity Advisories despite operator overlap)
- ❌ FS-ISAC (financial sector — commercial membership)
- ❌ H-ISAC (healthcare — commercial membership)
- ❌ MS-ISAC (state/local gov — public RSS available)
- ❌ Auto-ISAC, E-ISAC, others
- **Effort:** Public ISACs (MS-ISAC) are 1 PR each. Commercial ISACs require subscription and contractual data-redistribution review.

### E. GitHub Security Advisories (GHSA) — **NOT covered**

The customer specifically called this out. GHSA covers software-dependency
vulnerabilities at a granularity NVD doesn't (per-package, per-ecosystem:
npm, PyPI, Maven, RubyGems, Go modules, etc.).

- ❌ GHSA via the GraphQL API (`api.github.com/graphql`, `securityAdvisories` query)
- ❌ GHSA via the REST API (`api.github.com/advisories`)
- ❌ GHSA via Git-cloneable advisory database (`github.com/github/advisory-database`)
- **Effort:** 1 PR. Public, well-documented, no auth required for the REST endpoint at low volume. Authenticated GraphQL gives higher rate limit and richer fields.
- **Why it matters:** half of customer vulnerability exposure is in transitive npm/PyPI/Maven dependencies; NVD lags GHSA on these by days to weeks.

### F. Threat intelligence (commercial) — **NOT covered, by design**

- ❌ Mandiant (Google Cloud) — premium, contractual
- ❌ Recorded Future — premium
- ❌ Unit 42 (Palo Alto) — partly free RSS, mostly premium
- ❌ CrowdStrike Intelligence — premium, contractual
- **Effort:** Commercial sources are out of scope for a $50/mo brief unless we make a re-distribution deal.
- **Note:** Unit 42 and Mandiant both publish public-facing blogs as RSS — partial coverage of the strategic threat-actor narrative is accessible without paying. Worth one PR each.

### G. News aggregators — **partially covered**

- ✅ BleepingComputer, Krebs on Security, SANS ISC, The Hacker News, The Register
- ⚠️ Dark Reading — broken (403)
- ✅ SecurityWeek — works via feedburner redirect
- ❌ ZDNet Zero Day, Recorded Future blog, Schneier on Security blog
- **Effort:** ~1 PR per addition; not a priority — diminishing-return territory, news already over-represented vs. structured sources.

### H. AI/ML governance — **partially covered, weakly**

- ✅ `ai_governance_venturebeat` — general AI news, not specifically governance
- ✅ `ai_governance_mit_techreview` — same
- ❌ NIST AI Risk Management Framework updates (no dedicated RSS; release-driven)
- ❌ EU AI Act regulatory tracker
- ❌ OECD AI Policy Observatory
- ❌ MITRE ATLAS already covered (✅ in §2.1) — but ATLAS is attack TTPs, not governance
- ❌ AI Incident Database (`incidentdatabase.ai/api`)
- **Effort:** 1 PR for AI Incident Database; NIST AI RMF probably needs HTML scraping; EU AI Act tracker is several existing third-party trackers (Fairly AI, MLCommons) — pick one.

### I. Compliance / framework updates — **not covered**

- ❌ NIST CSF revision tracker
- ❌ ISO update notices (ISO 27001 series)
- ❌ PCI-DSS updates
- ❌ HIPAA enforcement / HHS OCR breach settlements (TODO already noted in `regulatoryEnforcementFeed.ts:17`)
- **Effort:** Most are slow-moving (annual to multi-year). Manual quarterly compilation by an analyst may be more reliable than a fragile scrape.

### J. Geopolitical / nation-state cyber — **partially covered via news**

- ✅ Krebs, BleepingComputer, The Register all carry nation-state coverage
- ❌ No dedicated source for Five Eyes joint advisories, CIS analysis, atlanticcouncil.org/cyber, or similar
- **Effort:** 1 PR each for RSS-based publishers; strategic content is mostly journalism rather than structured feeds.

## 5. Recommendations — top 3 sources to add

Ordered by value-to-effort ratio. Each is independently shippable.

### 1. GitHub Security Advisories (GHSA)

- **Source:** `https://api.github.com/advisories` (REST, public, no auth required for low volume)
- **Why:** Customer explicitly asked. NVD lags GHSA by days to weeks on
  software-dependency CVEs (npm/PyPI/Maven/Go). Half of typical enterprise
  vulnerability exposure is in transitive dependencies. GHSA is the only
  source with per-package granularity.
- **Effort:** 1 PR. Mirrors the NVD adapter pattern (`src/api/lib/nvdAdapter.ts`).
  Add slug `ghsa`, paginated GET, normalize to `signal_type='cve'`,
  `affected_vendor` from `package.ecosystem + package.name`.
- **New brief content:** ~50–200 net new advisories per week, weighted toward
  the language ecosystems the customer's vendors use.

### 2. Microsoft MSRC

- **Source:** `https://api.msrc.microsoft.com/cvrf/v3.0/updates` (CVRF JSON, public)
- **Why:** Windows + Office + Azure dominates enterprise patch surfaces.
  Patch Tuesday alone is the single largest predictable security event of the
  month for most CISOs. Currently we ingest Microsoft mentions only when
  BleepingComputer or Krebs writes about them — we lag the primary source.
- **Effort:** 1 PR. CVRF format is a published Microsoft standard. The
  monthly "updates" endpoint returns one entry per Patch Tuesday; per-update
  detail is a second API call. Add slug `psirt_msrc`.
- **New brief content:** ~10–30 advisories on Patch Tuesday week, sparse
  between. Rich CVE-to-product mapping for vendor-matching scoring.

### 3. CISA + FBI joint StopRansomware advisories

- **Source:** `https://www.cisa.gov/stopransomware/feed` (XML/RSS)
- **Why:** Named-actor, named-victim ransomware advisories are exactly the
  "executive must read this" content a $50/mo brief is sold against.
  Currently scattered through `cisa_alerts` (which buries them in generic ICS
  bulletins) and the news feeds. A dedicated channel surfaces them as a
  distinct content type.
- **Effort:** 1 PR. Reuses the `cisa_alerts` adapter pattern. Add slug
  `cisa_stopransomware`, `signal_type='threat_actor'`, severity defaulting
  to High.
- **New brief content:** ~2–8 advisories per week; high signal density.

### Honorable mentions, in order

4. **Cisco PSIRT** (`https://sec.cloudapps.cisco.com/security/center/psirtrss20/CiscoSecurityAdvisory.xml`) — same justification as MSRC for Cisco shops.
5. **Fortinet PSIRT** — high active-exploitation rate in 2024–2026; CISOs ask about it.
6. **AI Incident Database** (`https://incidentdatabase.ai/api`) — distinguishes AI governance from generic AI news; aligns with the platform's AI-system risk surface.

## 6. SQL queries for empirical grounding (run on staging)

These were not run during this audit — staging credentials are not present in
the codespace. Run them against staging to confirm the per-source weekly
volume assumptions in §2.4 and to identify any source slug surfacing in the
DB that this audit missed.

```sql
-- Per-source weekly signal count
SELECT source, COUNT(*) AS rows
FROM cyber_signals
WHERE ingestion_timestamp >= NOW() - INTERVAL '7 days'
GROUP BY source
ORDER BY rows DESC;
-- Catches: any slug producing 0 rows for 7 days (likely broken),
--          any slug surfacing in the DB but missing from §2 inventory.
```

```sql
-- Last successful fetch per source (confirms dead feeds vs. just-quiet)
SELECT source, MAX(ingestion_timestamp) AS last_signal
FROM cyber_signals
GROUP BY source
ORDER BY last_signal ASC NULLS FIRST;
-- Catches: every slug whose newest signal is older than ~3 days is
-- probably broken upstream — cross-check against §3.1 broken list.
```

```sql
-- Coverage of the 26-slug inventory (slugs in code that have never written a row)
WITH expected(slug) AS (VALUES
  ('cisa_kev'),('cisa_alerts'),('nvd'),('mitre_attack'),('mitre_atlas'),
  ('bleepingcomputer'),('krebsonsecurity'),('sans_isc'),('nist_news'),('ftc_news'),
  ('security_news_thehackernews'),('security_news_bleepingcomputer'),
  ('security_news_krebs'),('security_news_theregister'),
  ('regulatory_cisa'),('regulatory_nist'),('regulatory_ftc'),('regulatory_sec_8k'),
  ('regulatory_nydfs'),('regulatory_enisa'),('regulatory_ico'),('regulatory_fsb'),
  ('ai_governance_venturebeat'),('ai_governance_mit_techreview'),
  ('vendor_risk_securityweek'),('vendor_risk_darkreading')
)
SELECT e.slug, COALESCE(c.rows, 0) AS rows_ever
FROM expected e
LEFT JOIN (
  SELECT source, COUNT(*) AS rows FROM cyber_signals GROUP BY source
) c ON c.source = e.slug
ORDER BY rows_ever ASC;
-- Catches: any expected slug that has never produced a row — an immediate
-- broken-feed indicator independent of the URL health check in §3.
```

## 7. Summary

- **26 source slugs in code; 18 healthy, 7 broken (all RSS), 1 silent-fail on UA.**
- **Biggest content gap: zero PSIRT coverage.** The brief talks about patches
  without ingesting the patch publishers.
- **Operational gap: silent fail-open on broken feeds.** No surface signal
  when 27% of the input inventory dies. Add a synthetic monitor before
  growing the feed list.
- **Top 3 to add, in order: GHSA, Microsoft MSRC, CISA StopRansomware.** All
  RSS or simple JSON, all 1 PR each, all high-value executive content.

Recommended sequencing:

1. Fix the 7 broken URLs (1 PR; some replacements unknown so this is partly
   "remove dead slugs"). Add the SEC UA fix in the same PR.
2. Add a synthetic feed-health monitor (cron + alert) so the next dead URL
   surfaces in hours, not weeks.
3. GHSA adapter (1 PR).
4. MSRC adapter (1 PR).
5. CISA StopRansomware adapter (1 PR).

Out of scope for this audit: deciding whether the worker RSS pipeline and
the engine FeedAdapter pipeline should be unified (related to deferred
Issue B in `docs/dedup-audit.md`). That refactor is independent of source
coverage; pick it up when convenient.
